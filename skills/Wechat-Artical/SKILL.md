# 内容转公众号文章技能

将用户输入的内容转化为面向公众的文章。

## 核心定位

AI的角色是**创作者**：
```
接收用户在命令行输入的内容（主题、要点、素材等）
    ↓
理解整合 → 生成大纲 → 用户确认
    ↓
创建文章文件夹 → 拷贝素材图片（必须执行，禁止跳过）
    ↓
撰写文章 → 输出初稿 → 自我审视检查 → 修改文件 → 输出终稿
    ↓
输出2个文档：文章终稿 + 封面图提示词
    ↓
生成封面图 → 将封面图放入终稿 → 上传文章到微信公众号
```

## 输入方式

用户通过命令行直接输入以下内容：
- **文章主题**：明确文章要写什么
- **核心要点**：需要涵盖的关键观点或信息
- **补充素材**（可选）：相关数据、案例、背景等

**原则**：以用户输入的内容为主体，AI负责组织、扩展和润色。

## 工作流程

### 步骤1：理解用户输入并生成大纲

根据用户输入的内容，输出以下内容供用户确认：

```
## 内容理解与文章大纲

**核心内容**：[1-2句话概括用户输入的主旨]

**文章类型**：[观点输出/产品介绍/技术讲解/经验分享/其他]

**标题备选**：
1. xxx
2. xxx

**文章结构（无开头引言）**：

一、[章节标题]（约xx字）
   - 要点：xxx

二、[章节标题]（约xx字）
   - 要点：xxx

...

**计划扩展的内容**：
- [说明AI将如何扩展用户的要点]
```

**必须等待用户确认后才能开始写作。**

### 步骤2：创建文件夹并拷贝素材图片

═══════════════════════════════════════════════════════════════
⚠️ 【强制执行步骤】本步骤的所有命令必须实际执行，禁止跳过！
═══════════════════════════════════════════════════════════════

用户确认大纲后，**必须依次执行以下3个命令**：

**命令1：创建目录**
```bash
mkdir -p ./artical/文章名称/assets
```

**命令2：拷贝copy文件夹素材（必须执行！）**
```bash
cp -r copy/* ./artical/文章名称/assets/
```

**命令3：拷贝logo**
```bash
cp logo.png ./artical/文章名称/assets/
```

**【执行规则 - 必读！】**

1. **禁止预判**：不要猜测copy文件夹是否存在、是否有内容，直接执行命令
2. **禁止跳过**：即使你"觉得"copy文件夹可能为空，也必须执行命令
3. **禁止抑制错误**：不要使用 `2>/dev/null` 或其他方式屏蔽错误输出
4. **必须报告结果**：执行后告知用户命令执行结果（成功复制了哪些文件，或者报错信息）

**【正确行为示例】**
```
AI：我现在执行素材拷贝命令...

执行：cp -r copy/* ./artical/xxx/assets/

结果：成功复制了3个文件：img1.png, img2.jpg, diagram.png

（或者）

结果：copy文件夹不存在或为空，继续下一步。
```

**【错误行为示例 - 严禁！】**
```
AI：（心想：copy文件夹可能不存在，我跳过这步吧）
AI：已创建assets目录，现在开始写作...

↑ 这是错误的！你没有执行拷贝命令！
```

**将"文章名称"替换为实际的文章标题后执行。**

### 步骤3：撰写并输出初稿

按确认的大纲写作，遵循下方写作规范以及输出格式要求。
完成后将初稿保存至 `./artical/文章名称/artical.md`。

### 步骤4：自我审视与修改

对初稿从以下维度进行检查：

**A. AI痕迹检查**

**B. 逻辑合理性检查**

**C. 结构合理性检查**
- 开头摘要是否在250-350字之间？是否有吸引力？
- 各部分篇幅是否均衡？
- 结尾是否有力或有启发？
- 是否有冗余或可删减的部分？

**D. 图片位置是否放置正确性检查**

**E. 标题格式检查（重要！）**

检查每个章节标题是否符合以下格式：
```
【标题1】
# 第一个章节的标题

【标题2】
# 第二个章节的标题

【标题3】
# 第三个章节的标题
```

**常见错误（必须修正）**：
- ❌ `一、章节标题` → 应改为 `【标题1】` 换行 `# 章节标题`
- ❌ `【标题1】一、章节标题` → 标记和标题不能在同一行
- ❌ 只有`【标题1】`没有后续标题 → 每个章节都需要独立的【标题n】标记

**检查中若有错误，直接修改 `artical.md` 文件。**

### 步骤5：输出终稿及附属文档

修改初稿为终稿，并完成附属文档，保存至 `./artical/文章名称/`：

**A. 文章终稿** (`artical.md`)
- 修改后的完整文章
- 增加200字的引言

**B. 封面图设计说明** (`cover_design.md`)
- 提供1封面图生成提示词。

**封面图提示词编写原则**：
- 用生活/工作中的具象场景隐喻文章所表达的抽象概念
- 一眼能理解主题，不需要解释
- 避免科技感过重的抽象画面
- 参考风格：指挥家指挥AI乐团、园丁浇灌AI应用等
- 需要使用现实的照片风格来生成封面图，除了清晰的图片内容描述外，还需要提供大量反向词来保证图片的质量，图片的尺寸是16:9横屏。这个要求也要写到封面图提示词中，提示词使用英文进行编写。

### 步骤6：生成文章封面图
运行命令：`python /opt/homebrew/lib/node_modules/clawdbot/skills/Wechat-Artical/scripts/img_creator.py ./artical/文章名称文件夹`

> 注意：使用脚本的绝对路径，以便从任意工作目录执行

### 步骤7：修改终稿，将得到的文章封面图路径放在logo图路径下面，放在对应的位置，并提供标注【封面主图】

### 步骤8：上传文章到微信公众号
运行命令：`python /opt/homebrew/lib/node_modules/clawdbot/skills/Wechat-Artical/scripts/wechat_publisher.py ./artical/文章名称文件夹`

> 注意：使用脚本的绝对路径，以便从任意工作目录执行

## 写作规范

### 创作原则

用户输入 → 公众文章，需完成以下创作：

| 维度 | 用户输入特点 | 文章要求 |
|------|----------|----------|
| 语言 | 可能是口语、要点式 | 书面、自成体系 |
| 结构 | 可能零散、跳跃 | 有逻辑、递进 |
| 受众 | 用户了解背景 | 读者是小白 |

**创作示例**：
```
用户输入："AI工具太复杂，用户还要学提示词、搭工作流"

文章表达："当前很多AI工具要求用户学习提示词、搭建工作流，这实际上是把技术复杂性转嫁给了用户。"
```

### md文档格式要求（重要！）

```md
【文章标题】文章标题

【logo】
![logo2-un](assets/logo.png)

【引言】
文章引言内容（200字左右，纯文本，不要任何md格式）

【封面主图】
![cover](assets/cover.png)

【标题1】
# 第一章节的标题

第一章节的正文内容...

【标题2】
# 第二章节的标题

第二章节的正文内容...

【标题3】
# 第三章节的标题

第三章节的正文内容...

（以此类推，每个章节都需要【标题n】标记）
```

### 标题格式详细说明（必读！）

⚠️ **每个章节标题必须遵循以下格式**：

```
【标题n】
# 章节标题文字
```

**格式要求**：
1. `【标题n】` 必须单独占一行（n为数字：1、2、3...）
2. `【标题n】` 后面必须换行
3. 章节标题文字在下一行，前面加 `#` 符号
4. 不要使用"一、"、"二、"这种中文数字格式

**✅ 正确示例**：
```
【标题1】
# 为什么值得尝试写小说

写小说，可能是门槛最低、回报最高的自我表达方式...

【标题2】
# 小说创作的核心三要素

一篇小说无论长短，都离不开三个核心要素...

【标题3】
# 如何找到并打磨你的故事创意

好的故事创意，往往藏在三个地方...
```

**❌ 错误示例**：
```
一、为什么值得尝试写小说        ← 错！缺少【标题1】标记

【标题1】一、为什么值得尝试写小说  ← 错！标记和标题在同一行

【标题1】
为什么值得尝试写小说            ← 错！标题缺少 # 符号
```

### 其他格式说明

- logo.png 和素材图片在步骤2中已拷贝到assets目录
- 在 md 文档中引用logo：`![logo2-un](assets/logo.png)`
- 封面主图由 img_creator.py 生成后自动放入 assets 目录
- 引言部分禁止任何 md 格式，纯文本即可
- 正文可以使用：一级标题(#)、加粗(**)、无序列表(-)，禁止其他样式

### 观点呈现

用户输入的内容可以三种方式呈现：

1. **直接引用**：精炼有力的原话，用引号保留
2. **转述融入**：改写后融入正文
3. **展开论证**：以观点为起点，AI进行扩展论述

### 基本要求

- **字数**：1500-2200字
- **开头引言**：文章开头需有一个200字左右的引言段落，作为全文摘要，需具备吸引力，让读者有继续阅读的欲望【开头引言需要在完成所有其他内容的编写后，在完成终稿阶段进行编写，而不是在初稿阶段编写；开头引言不要提供标题，直接是一个文本段落即可】
- **段落**：每段3-5句，避免大段文字
- **列表**：仅在分类、步骤等必要时使用，正文以段落为主
- **图片**：按用户提供的路径，使用 `![描述](路径)` 格式插入
- 即使用户输入中提到了某些品牌，也不要直接说出来，而是以"某些"产品作为替代

### 语言风格

- **小白友好**：专业术语首次出现时用大白话解释
- **理性克制**：陈述事实、解释逻辑，避免煽情或过度营销
- **通俗易懂**：可用类比帮助理解复杂概念

**技术解释示例**：
```
Agent（智能体）可以理解为能自主完成任务的AI助手——你告诉它目标，它会自己规划步骤并执行。
```

## 注意事项

1. **以用户输入为核心**：文章主体内容必须基于用户提供的主题和要点
2. **不自行搜索**：严格基于用户输入的内容进行创作
3. **大纲必须确认**：等用户确认后才开始写作
4. **素材拷贝必须执行**：步骤2的拷贝命令必须实际执行，禁止预判跳过，执行后报告结果
5. **口语转书面**：如果用户输入是口语化的，需转化为书面语言
6. **补充背景**：用户输入中省略的背景信息需在文章中补全
7. **灵活组织**：根据内容特点自行决定文章结构，不拘泥于固定模板
8. **标题格式必须正确**：每个章节都要有【标题n】标记，标题用 # 格式
9. **自我审视**：完成初稿后必须进行审视修改，消除AI痕迹、修正逻辑和结构问题
